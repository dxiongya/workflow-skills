# 建议池（Proposal Backlog）

## 核心理念

每个团队成员在执行任务过程中都会产生洞察——发现可以做得更好的地方、缺失的功能、体验上的痛点、技术上的改进机会。这些不应该丢失，也不应该打断当前工作，而是收集起来，在合适的时机评审。

## 谁可以提建议

**所有角色**，在任何环节：

| 角色 | 典型场景 |
|------|---------|
| **Dev** | 开发过程中发现代码结构可优化、缺少某个功能会更方便用户、API 设计有改进空间 |
| **QA** | 测试过程中发现用户体验不顺畅、缺少引导、某个流程可以简化、安全策略可以加强 |
| **Design** | 审查过程中发现交互模式可优化、视觉一致性可提升、新的设计趋势值得跟进 |
| **PM** | 用户反馈、竞品观察、数据分析中发现的机会点 |

## 建议格式

```
## 建议: [简明标题]
提出人: [角色 + 名称]
提出时间: [日期]
发现场景: [在做什么的时候发现的]

### 描述
[详细说明建议内容]

### 动机
[为什么觉得这个值得做——用户痛点、效率提升、风险规避等]

### 参考
[截图、数据、竞品链接等佐证材料（如有）]
```

## 建议的生命周期

```
提出 → 进入建议池 → 下一迭代拆卡时评审 → 采纳/拒绝/延后
```

### 提出
- 执行任务过程中随时记录
- 不打断当前工作——先记下来，不需要深入分析
- 建议和当前卡片的 bug 是不同的东西：bug 是"不符合需求"，建议是"可以做得更好"

### 进入建议池
- 按阶段归集（Sprint N 的建议池）
- 不做优先级排序——这是评审时的事

### 评审（每个阶段拆卡时）
PM + Design 主导评审，Dev Lead 参与技术可行性判断：

| 评审结果 | 说明 |
|---------|------|
| **采纳** | 符合产品方向，拆为下一阶段的任务卡 |
| **延后** | 有价值但当前优先级不够，保留到未来迭代 |
| **拒绝** | 不符合产品方向或投入产出比不合理 |

### 评审规则
- PM + Design 有最终决定权
- **采纳标准**：符合产品定位和用户需求方向
- 拒绝时必须说明理由，让提出人理解决策逻辑
- 延后的建议不会消失，每次拆卡时重新评估

## 建议 vs Bug vs 需求

| 类型 | 定义 | 谁决定 | 去向 |
|------|------|--------|------|
| **Bug** | 当前功能不符合验收标准 | QA 发现，Dev 修复 | 当前阶段 bug 卡 |
| **建议** | 当前功能可以做得更好 | 任何人提出，PM+Design 评审 | 建议池 → 未来迭代 |
| **需求** | 产品规划中的功能 | PM 定义 | 当前/未来阶段任务卡 |

关键区分：
- "按钮点击没反应" → **Bug**（不符合验收标准，立即修）
- "按钮点击后加个动画反馈会更好" → **建议**（体验优化，进建议池）
- "需要增加导出 PDF 功能" → **需求**（新功能，PM 规划）

## 鼓励提出建议

- 建议被拒绝不代表建议不好——可能只是时机或优先级的问题
- 高质量的建议应该有具体场景和动机，不是空泛的"可以更好"
- 每个阶段结束时可以回顾：哪些建议被采纳了，效果如何
