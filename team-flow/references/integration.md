# 集成测试（阶段收尾）

## 触发条件
当前阶段**所有任务卡**状态为 **COMPLETE**

## 参与角色
- **QA**：主导测试、输出 bug 报告
- **Dev Lead**：bug 优先级确认、分配修复
- **PM + Design**：需求争议仲裁

## 目标
对阶段交付的完整功能进行全面回归测试，发现单卡测试遗漏的问题。

## 测试范围

| 维度 | 说明 |
|------|------|
| **功能完整性** | 所有卡片的功能是否正常工作 |
| **功能交叉** | 不同功能之间是否冲突 |
| **数据一致性** | 跨模块的数据流是否正确 |
| **性能** | 整体加载、响应是否在可接受范围 |
| **兼容性** | 不同设备/浏览器/系统版本 |
| **安全** | 权限、数据保护、输入校验 |

## 流程

### 1. QA 全面测试
- 使用 debug-kit 对所有功能模块逐一验证
- 重点关注卡片交叉区域（A 卡的改动是否影响了 B 的功能）
- 记录所有问题

### 2. 输出 Bug 报告

```
## 集成测试 Bug 报告 — [阶段名称]
测试人：[QA]
测试日期：[日期]

### Bug 列表
| # | 描述 | 严重程度 | 关联卡片 | 截图 |
|---|------|---------|---------|------|
| 1 | ... | P0 | CARD-012 | ... |
| 2 | ... | P1 | CARD-005 | ... |
| 3 | ... | P2 | 无关联 | ... |

### 统计
- P0: N 个
- P1: N 个
- P2: N 个
- P3: N 个
```

### 3. Bug 卡创建与分配

QA + Dev Lead 共同操作：

1. **确认是 bug**：排除假阳性
   - 不确定的问题 → 拉 PM + Design 讨论，明确是需求还是 bug
   - 确认后才创建 bug 卡
2. **创建 bug 卡**：复现步骤、预期/实际结果、截图、严重程度
3. **分配执行者**：Dev Lead 根据改动归属分配给对应开发

### 4. Bug 修复

```
Dev Lead 分配 → Dev 修复 → Dev 自测 → QA 复验
                                        ↓
                                  通过 → 关闭
                                  不通过 → 继续修复
```

### 5. Bug 驳回机制

执行开发有权驳回 bug 卡，但必须：
- 说明为什么不是 bug（预期行为、需求定义如此等）
- 提供证据（需求文档、设计稿、技术约束）

驳回审查流程：
```
Dev 驳回 → Dev Lead + PM + Design + QA 四方审查
                ↓                    ↓
          确认不是 bug → 关闭卡片    确认是 bug → Dev 继续修复
```

## 阶段完成标志

- 所有集成测试 bug 卡为 **COMPLETE** 或被审查关闭
- P0/P1 bug 全部修复
- P2 bug 已评估（修复或标记延后）
- QA 确认签收

## 进入下一阶段

阶段收尾完成后，开始下一阶段的**拆卡**，循环继续。
