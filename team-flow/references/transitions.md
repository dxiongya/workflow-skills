# 状态转移规则（State Transition Gates）

**这是 team-flow 的执行核心。** 每次卡片状态变更前，必须通过对应关卡的所有检查项。

## 执行要求

每次状态转移时，AI 必须：
1. **列出当前关卡的所有检查项**
2. **逐条回答每个检查项的结果**（通过/未通过，附证据）
3. **如有未通过项，停止转移**，说明原因并处理
4. **所有检查项通过后，向用户报告转移原因**，请求确认
5. **用户确认后才执行状态变更**

---

## 卡片分类（决定验证方式）

每张卡在拆卡阶段必须标注分类，③ 和 ④ 关卡按分类使用不同的证据形式。

| 分类 | 定义 | ③ 自测证据要求 |
|------|------|---------------|
| **visual** | 改变用户看到的像素（UI、布局、文字、颜色） | **必须** debug-kit screenshot 对比 + 结构验证 |
| **behavioral** | 改变运行时行为（数据流、事件、API 调用） | debug-kit 交互日志 + console/log 检查 |
| **structural** | 只改基础设施、类型、配置、未被引用的新文件 | **允许豁免 screenshot**，用 grep / import graph / compile 输出代替 |

### 零影响卡（structural 特例）

如果卡片满足以下全部条件，视为"零影响"，`UI 视觉无变化`类验收标准可用结构性证据代替截图：

- 新增的代码/文件**尚无任何组件引用**（grep 证实引用数为 0）
- 未修改任何已被引用的符号/文件
- 未改动 build 配置或运行时入口

证据格式示例：
```
验收标准: "UI 视觉无任何变化"
  → 通过 ✓ 结构性证据:
     grep:className.*<new-class>@src/components/ → 0 文件
     grep:import.*<new-module>@src/              → 0 文件
     diff:index.css → 无修改
```

---

## 证据语法

所有 checkpoint 的验证结果**必须**用下列语法之一表达，便于扫读与复核：

| 语法 | 用途 | 示例 |
|------|------|------|
| `file:<path>` | 文件存在性 | `file:design/tokens.css` |
| `grep:<pattern>@<path>` | 代码搜索 | `grep:var\(--cb-@src/components → 0 匹配` |
| `compile:<cmd>` | 编译 / 类型检查 | `compile:pnpm tauri dev → Finished dev profile` |
| `screenshot:<path>` | 视觉对比 | `screenshot:/tmp/after.png`（附说明） |
| `log:<path>:<line>` | 运行时日志 | `log:/tmp/dev.out:142 → WebSocket listening` |
| `test:<cmd>` | 测试运行结果 | `test:pnpm test → 42 passed` |
| `diff:<path>` | 文件差异 | `diff:index.css → 无修改` |

混合使用时，一行一条。模糊描述如"已验证"、"OK"、"差不多了"**不接受**。

### 转移报告里的证据块示例

```
验收标准 1: "design/components.css 存在，13 语义类齐全，零硬编码"
  → 通过 ✓
  → file:design/components.css
  → grep:^\.[a-z-]+ @design/components.css → 45 条 rule headers
  → grep:#[0-9a-fA-F]{3,8}@design/components.css → 0 匹配

验收标准 2: "pnpm tauri dev 编译成功"
  → 通过 ✓
  → compile:pnpm tauri dev → Finished dev profile, binary running
  → log:/tmp/byd6mz9x4.output:tail → 无 error
```

## 批量转移规则

**初始拆卡场景下的 PLAN → TODO 允许批量**：当一批卡片在同一次拆卡会议中同时就绪时，可以对"整批"做一次关卡检查，在一份报告里列明"本批 N 张卡全部满足条件"，而不必逐卡重复。

其它转移（TODO→DEVING、DEVING→DEV DONE、DEV DONE→QA 等）**仍必须逐卡单独进行**，因为每张卡的上下文、进度、证据都不同。

批量转移的报告模板：

```
## 批量状态转移: <阶段名> PLAN → TODO（N 张卡）

### 关卡检查（对整批生效）
- [x] 所有卡片标题清晰
- [x] 所有卡片验收标准可验证（抽查：CARD-01 / CARD-05 / CARD-08）
- [x] 所有卡片设计引用齐全（抽查同上）
- [x] 依赖图无环
- [x] 三方无异议

### 批次清单
| ID | 标题 | 分类 | 优先级 |
| … | … | … | … |

### 确认
是否批准本批 N 张卡从 PLAN 转为 TODO？
```

## ① PLAN → TODO（拆卡完成）

### 关卡：卡片就绪检查

转移前必须逐条确认：

```
[ ] 卡片标题是否清晰描述了要做什么？
[ ] 任务需求是否具体到可执行的程度？
[ ] 交互方式是否已定义（用户怎么操作、系统怎么响应）？
[ ] 验收标准是否可验证（不是模糊描述，而是可用 debug-kit 验证的条件）？
[ ] 相关资源是否齐全（设计稿链接、API 文档、依赖说明）？
[ ] 优先级是否已确定（P0/P1/P2/P3）？
[ ] 执行者是否已分配？
[ ] PM、Design、Dev 三方是否对卡片内容无异议？
```

### 谁确认
PM 确认所有字段齐全 → 批量转为 TODO

### 不满足时
缺哪项补哪项，不允许带着空白字段进入 TODO。

---

## ② TODO → DEVING（开卡）

### 关卡：上下文对齐检查

转移前必须逐条确认：

```
[ ] PM 是否已向 Dev 说明了需求背景和目标？
[ ] Design 是否已向 Dev 说明了交互细节和设计规范？
[ ] Dev 是否已确认技术可行性？
[ ] Dev 对卡片内容是否还有疑问？（有疑问则不转移）
[ ] Dev 是否已理解所有验收标准？
[ ] 是否存在依赖其他未完成卡片的阻塞？（有则转 BLOCK）
```

### 谁确认
Dev 确认"我理解需求，没有疑问，可以开始"

### 不满足时
疑问未解决 → 继续讨论；有依赖 → 转 BLOCK。

---

## ③ DEVING → DEV DONE（开发自测通过）

### 关卡：自测验证检查

这是 Dev 对自己工作的验证。**必须使用 debug-kit 产出证据。**

```
[ ] 功能是否已全部实现？（对照卡片需求逐条确认）
[ ] 验收标准是否逐条通过？（每条标准附 debug-kit 截图或日志证据）
[ ] UI 是否与设计稿一致？（debug-kit screenshot 截图对比）
[ ] 交互是否正确？（debug-kit tap/type 操作验证）
[ ] 控制台/日志是否无异常错误？（debug-kit console/log 检查）
[ ] 边界情况是否处理？（空数据、异常输入、长文本）
```

### 谁确认
Dev 自己确认，但必须**展示证据**（截图路径、日志输出）

### 不满足时
哪项没过就继续开发，不允许"差不多了先提交"。

### 证据格式
```
验收标准 1: "点击提交后显示成功提示"
  → 通过 ✓ 截图: /tmp/self-test-submit.png
  → debug-kit 命令: bash $P/ios-ctl.sh tap 195 400 && bash $P/ios-ctl.sh screenshot /tmp/self-test-submit.png

验收标准 2: "输入为空时提交按钮禁用"
  → 通过 ✓ 截图: /tmp/self-test-empty.png

控制台检查:
  → 通过 ✓ 无错误输出（bash $P/ios-ctl.sh log 5）
```

---

## ④ DEV DONE → QA（结卡通过）

### 关卡：团队验收检查

这是整个流程中最重要的关卡。**三个角色各自有独立的验收维度。**

#### PM 验收
```
[ ] 功能是否满足卡片描述的需求？
[ ] 验收标准是否逐条满足？
[ ] 是否有需求遗漏或理解偏差？
```

#### Design 验收（必须使用 debug-kit 截图）
```
[ ] 布局是否与设计稿一致？（截图对比）
[ ] 颜色是否使用 DESIGN.md 定义的 token？
[ ] 字体大小、字重、行高是否正确？
[ ] 交互状态（hover/active/disabled）是否实现？
[ ] 边界 UI 状态（空数据/加载/错误）是否处理？
[ ] 响应式适配是否正确？（如适用）
```

#### Dev 演示
```
[ ] Dev 是否演示了实现结果？
[ ] Dev 是否说明了技术方案和注意事项？
```

### 结果判定

| 结果 | 条件 | 后续 |
|------|------|------|
| **通过** | PM + Design + Dev 三方全部确认通过 | → QA |
| **不通过** | 任何一方有未通过项 | 状态回退到 DEVING，在原卡继续修改 |

### 关键规则
- **不通过时不创建新卡**，在原卡上继续
- **三方中任何一方不通过就是不通过**，不允许"先过了再说"
- 设计问题和功能问题都必须在结卡阶段解决

---

## ⑤ QA → QA DONE（QA 验收通过）

### 关卡：测试完成检查

```
[ ] 是否已完成功能逻辑测试？（正向+反向+边界）
[ ] 是否已完成 UI 测试？（对照 DESIGN.md）
[ ] 是否已完成交互测试？（操作反馈验证）
[ ] 是否已完成安全检查？（XSS/注入/敏感数据）
[ ] 是否已输出完整测试报告？
[ ] 发现的所有 bug 是否已创建 bug 卡？
[ ] 所有 bug 卡是否已修复并复验通过？
[ ] 测试报告结论是否为"通过"？
```

### 谁确认
QA 确认所有测试通过，所有 bug 卡已关闭

### 不满足时
- 有 bug 未修复 → 原卡保持 QA 状态，等待 bug 卡关闭
- 测试未完成 → 继续测试，不允许部分通过

---

## ⑥ QA DONE → COMPLETE（完成确认）

### 关卡：最终确认

```
[ ] QA 测试报告是否已归档？
[ ] 所有关联的 bug 卡是否状态为 COMPLETE？
[ ] 是否有遗留的已知问题需要记录？
```

### 谁确认
QA 签收

---

## 任意 → BLOCK（阻塞）

### 关卡：阻塞声明

```
[ ] 阻塞原因是什么？（具体描述）
[ ] 依赖什么才能解除？（其他卡片/外部资源/需求澄清）
[ ] 是否已通知相关方？
```

### 谁触发
任何角色都可以标记 BLOCK

### 典型场景示例

**场景 A — 依赖其他未完成的卡**
```
卡片: REFACTOR-05 (Editor 重写)
当前状态: DEVING
触发原因: 需要 REFACTOR-03 (Shell 重写) 定义的 `.tab` 语义类，
         但 REFACTOR-03 还在 DEVING，尚未提交 components.css 里的 .tab 定义。
依赖解除条件: REFACTOR-03 推进到 DEV DONE 或更高状态
已通知: Dev Lead（由他决定是否并行推进或等待）
```

**场景 B — 等待外部资源**
```
卡片: FEAT-12 (接入支付回调)
当前状态: TODO
触发原因: 支付网关 Webhook 文档 v2 尚未发布，当前 v1 文档里回调字段定义与实际不符
依赖解除条件: 收到供应商的 v2 文档 或 拿到沙箱环境的真实回调样本
已通知: PM（由 PM 去跟供应商催）
```

**场景 C — 需求澄清**
```
卡片: BUG-031 (会话列表排序错乱)
当前状态: DEVING
触发原因: 发现两种合理排序（按 updated_at / 按 last_message_at），
         需求文档没说，两种实现对用户体验影响不同
依赖解除条件: PM 给出明确决策
已通知: PM（已在卡片评论区 at）
```

**场景 D — 技术方案重估**
```
卡片: FEAT-08 (实时协作光标)
当前状态: DEVING
触发原因: 自测发现当前方案（HTTP polling）延迟 > 2s，不满足"近实时"验收标准。
         需要 Dev Lead 决定是否改用 WebSocket
依赖解除条件: 技术方案重新评估后的决定
已通知: Dev Lead
```

---

## BLOCK → 原状态（解除阻塞）

### 关卡：解除确认

```
[ ] 阻塞原因是否已解决？（附证据）
[ ] 解决方案是否影响了卡片的需求或验收标准？（如有变更需更新卡片）
[ ] 卡片回到哪个状态？（原状态或更早状态）
```

### 回退规则

- **默认回原状态**：解除后回到 BLOCK 前的状态（如原本 DEVING → BLOCK，解除后回到 DEVING）
- **如果解决方案改变了需求/验收标准**：必须把卡片**回退到 TODO**，重新开卡确认上下文
- **如果是技术方案重大调整**：Dev Lead 评估后可能需要重新拆卡，原卡作废

---

## 非法转移

以下转移是**禁止的**，AI 不得执行：

| 非法转移 | 原因 |
|---------|------|
| PLAN → DEVING | 跳过了开卡环节，Dev 没有对齐上下文 |
| TODO → DEV DONE | 跳过了开发过程 |
| DEVING → QA | 跳过了自测和结卡，没有验收 |
| DEV DONE → COMPLETE | 跳过了 QA 测试 |
| QA → COMPLETE | 跳过了 QA DONE 确认 |
| 任意 → COMPLETE | 只能从 QA DONE 转入 |

如果 AI 发现自己想要执行以上任何转移，必须**停止并提醒用户**流程不正确。

## 转移报告模板

每次状态转移时，AI 向用户输出：

```
## 状态转移: [卡片名] [当前状态] → [目标状态]

### 关卡检查
- [x] 检查项 1: 通过（证据：...）
- [x] 检查项 2: 通过（证据：...）
- [x] 检查项 3: 通过

### 转移原因
[简要说明为什么可以转移]

### 确认
是否同意将卡片状态从 [当前] 转为 [目标]？
```
